# Tier 3 前沿模式(難題專用)

> 定位:編排是中等任務的槓桿、難題的稅。真難題要的是**一個深 context 燒到底**,
> 不是切碎給多個淺 context。Opus 4.8 每 token 半價於 Fable —— 同樣的錢燒兩倍深度,
> 這是 Tier 3 的成本數學基礎。

## 觸發判準(符合任一)

- 新問題,無既有模式可循(LESSONS/DECISIONS 幫不上)
- 預期單 session 數小時的推理/建造
- 步驟高度耦合、切不開 —— fable-emu 的計畫階段若判 complexity=complex 且步驟間強相依,是「編排稅最重」的訊號,應改走本模式
- 時間信封在 Tier 3 **明文放寬**:難題本來就以小時計,Fable 也一樣;費用信封仍成立(≤ Fable 同任務)

## 模式 A:深潛(預設)

單一 Opus 4.8 session、`effort: xhigh`(正確性壓倒成本時用 `max`)、完整任務規格一次給足、
headless 時掛 `fable-run.ps1` 監督。

**提詞模板(去指令化 —— 難題入口要留白,kit 的詳盡條款在此讓路)**:

```text
我在做[更大的目標],為了[誰/什麼]。他們需要[產出能達成什麼]。

難題:[完整描述,一次給足——背景、已知、已試過什麼]

約束:[真正的硬約束,只列不可違反的]
驗收:[完成的客觀定義——測試、基準、可觀察行為]
記憶:動工前讀 .fable/LESSONS.md;過程中的教訓隨手寫回。

方式由你決定。建立自我檢查的方法,按固定間隔用 fresh-context subagent 對照驗收標準核驗。
```

注意模板裡**沒有**階段契約、沒有選項義務、沒有警報表——那些是給中等任務與弱模型執行層的。
唯二保留:記憶迴路與 fresh-context 自驗(官方指南對 Fable 級模型的同款建議)。

## 模式 B:競試(僅限「有客觀驗證器」的難題)

`Workflow({name: "deep-attempts", args: {task, verify, n: 2}})`

- 原理:測試時間運算的鐵律——**有驗證器時,廣度可以買回深度**。N 個互不知情的獨立深潛,
  裁判到各 worktree 親自重跑驗證(不信自報)擇優。
- 成本:2 個 Opus xhigh 深潛 ≈ 1 個 Fable 深潛的錢;3 個 = 1.5 倍,僅在極高價值任務用。
- **verify 必填是硬性設計**:沒有客觀驗證器,裁判就需要 Fable 級品味,循環論證——那種題直接走深潛或上 Fable。

## 誠實邊界(Tier 3 也買不回的)

- **無驗證器的難題**(新穎架構品味、開放式研究方向):best-of-N 需要裁判,裁判需要的正是缺的那個判斷。
- **多日級長程自主**:跨 compaction 的指令保持是權重能力,effort 和編排都不影響。
- 這兩類是 Fable 的專屬地形;kit 的記憶/留痕系統與 Fable session 共用同一套知識庫,雙軌並用。
