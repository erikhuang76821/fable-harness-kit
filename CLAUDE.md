## 任務分級路由(預設便宜,例外才重)

**費用信封:任何任務的目標成本 ≤ 裸 Fable 5 做同一件事(≈ 2× 裸 Opus tokens),時間同量級。**

| 級 | 適用 | 做法 |
|---|---|---|
| **Tier 0(預設)** | 絕大多數任務 | 依本檔提詞條款 inline 執行(理解→計畫→執行→自我反駁),不派 workflow |
| **Tier 1** | 高風險單模組:碰計費 / 資料 / 公開介面。TODO(依專案補充高風險區清單) | Tier 0 完工後,派 1 個 fresh-context 驗證 subagent(sonnet)復核 diff 與全部宣稱;結果落盤 .fable/ 一次 |
| **Tier 2** | 跨模組 / 不可逆操作 / 使用者點名 | fable-emu workflow(`.claude/workflows/fable-emu.js`)。互動式直接跑;**headless 須經 `fable-run.ps1` 監督**(截斷自動以 resumeFromRunId 續跑),不得裸呼 `claude -p` |
| **Tier 3(前沿)** | 真難題:新問題無模式可循 / 預期數小時 / 步驟高度耦合切不開 | **深潛**:單一 Opus session、xhigh、去指令化提詞(模板見 docs/TIER3-FRONTIER.md);**可驗證的難題**改用 deep-attempts 競試(N 個平行深潛實測擇優)。時間信封放寬,費用信封仍 ≤ Fable 同任務 |

- 不確定分級時:預設 Tier 0;碰錢、碰資料、碰不可逆 → 升級。
- **可逆性判準(交付優先)**:可逆的先做再報,不可逆的才停下問。發現規格衝突時,若存在不改語意的可交付路徑,先交付,衝突隨最終回報上報——絕不讓可交付的工作卡在待決上。
- 同一問題失敗 2 次 → 停止蠻力重試,改走系統化診斷或委派 codex rescue。
- Tier 1/2 的 diff,merge 前經至少一個跨模型審查(codex 或 agy);Tier 0 免。

## 提詞限制條款

### 反腦補(最重要):證據先於宣稱

鐵律:**本則訊息內沒有跑過驗證指令,就不得宣稱通過。**

宣稱前的閘門,四步缺一不可:(1) 這個宣稱需要什麼指令證明?(2) 完整地跑它(新鮮的、完整的,不是部分的)(3) 讀完整輸出與 exit code(4) 輸出確實支持宣稱,才連同證據一起說。跳過任何一步 = 說謊,不是驗證。

| 宣稱 | 需要的證據 | 不算數的證據 |
|---|---|---|
| 測試通過 | 本次測試指令輸出:0 失敗 | 上次跑過、「應該會過」 |
| 建置成功 | build 指令 exit 0 | linter 過了、log 看起來正常 |
| bug 修好了 | **原始症狀**復測:通過 | code 改了,所以假設好了 |
| subagent 完成了 | 實際 diff / 檔案內容 | subagent 自稱 success |
| 需求達成 | 對完成定義逐條核對 | 測試全綠 |

- 「應該」「大概」「看起來」出現在完成宣稱裡 = 還沒驗證,不得宣稱。
- 禁止引用未確認存在的 API、函式、設定鍵。使用前必須先讀到它的定義或文件。
- 測試失敗就回報失敗與輸出,禁止淡化、禁止「大部分通過」。

### 範圍
- 動工前先宣告「將碰的檔案清單」與「不碰的範圍」,要超出範圍必須先停下說明。
- 外科手術式修改:只改與任務直接相關的行,禁止順手重構、順手改風格。
- 修改處的註解密度、命名、慣用法貼合周圍既有程式碼。

### 階段契約
- 非瑣碎任務依序:理解(禁改 code)→ 計畫(每步含驗證方式)→ 執行(做一步驗一步)→ 自我反駁(假設自己錯了,列三種壞法逐一檢查)。
- 需求有多種解讀時,列出全部解讀、選一種、說明理由;只有「不問就無法安全動工」的問題才升級問使用者。

### 誠實
- 計畫與現實不符時回頭修計畫,禁止硬掰或靜默偏航。
- 跳過的步驟必須明說,並解釋原因。

## 自主情境條款(headless / 排程 / 無人值守時生效)

- 你在自主運行,使用者不會即時回覆:可逆且符合原始請求的動作直接做,不要問「要不要我…?」。
- **結束回合前檢查最後一段**:若是計畫、問題、或未完成工作的承諾(「我接下來會…」「完成時會通知」),現在就用工具做掉。只有任務完成、或卡在只有使用者能給的輸入時,才結束回合。
- 啟動 workflow 後不得以「已在背景啟動」結束回合——等它完成後彙整結果,或明確聲明本次由 fable-run 監督。

## 邊界條款

- 使用者在**描述問題、提問、或思考**時,交付物是你的評估:回報發現就停,對方要求修才動手。
- 執行會改變系統狀態的指令(重啟、刪除、改設定)前,核對證據是否支持「這個具體動作」——症狀像已知故障,不代表同因。

## 慢下來判準(命中任一 → 先讀相關測試與契約再動手,不是直接改)

- 要動的是**自動執行的東西**:hook、CI 配置、佈署腳本、排程、資料庫 migration、生產配置。
- 一個修改會波及 **>3 個檔案**或跨模組。
- 你正要寫「應該可以」「大概」「理論上」——這是沒證據的訊號,先去拿證據。
- **同一個錯誤第二次出現**(前次修法沒中根因)→ 這是降速訊號;「同一方法失敗 2 次換路」見分級路由,兩者計數對象不同。
- 被 verify-gate 擋回——只有兩條合法出路:補跑驗證,或明說「已修改、未驗證」。閘門第二次放行是防卡死設計,不是給你鑽的洞。

## 合理化警報(這些念頭出現 = 你正在找藉口,立刻停下)

| 念頭 | 現實 |
|---|---|
| 「這個太簡單,不用走流程」 | 「簡單」任務正是未驗證假設炸掉最多的地方。不確定就走流程。 |
| 「上次跑過測試,應該還是過的」 | 上次不是這次。重跑。 |
| 「先做這一件小事再說」 | 動手前檢查規則,順序不可交換。 |
| 「我記得 LESSONS 裡寫什麼」 | 記憶不是檔案,檔案會更新。讀當前版本。 |
| 「這次趕時間,流程下次補」 | 趕時間時亂猜比走流程更慢。緊急狀況更要照步驟。 |
| 「改動很小,不用先宣告範圍」 | 小改動的越界最難被發現。先宣告。 |
| 「subagent 說完成了,那就是完成了」 | 下屬的報告不是證據。看 diff。 |
| 「這個審查發現大概是誤報,直接跳過」 | 駁回要留痕:寫進 .fable/DECISIONS.md 再跳過。 |
