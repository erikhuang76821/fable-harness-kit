# Benchmark PROTOCOL v2(難題帶;預註冊,動手前寫死——修改需獨立 commit)

> 承 PROTOCOL.md 與消融追補的結論:v1 難度帶三臂全飽和,kit 相對裸 Opus 的增量未被測量。
> v2 的唯一問題:**在有鑑別力的任務帶,kit(路由+workflow+hooks)相對裸 Opus 有沒有增量?**

## 假設

H2:kit 化 Opus 在「跨模組回歸地雷 / 曖昧規格 / 修一處不破三處」任務帶的
密封驗收通過率 > 裸 Opus。
H0:兩臂通過率無差(kit 的增量只是保險與留痕,不含此帶的正確率)。

## 關鍵設計:密封驗收(sealed tests)

- fixture 分兩層:**shipped**(模型可見:部分測試、規格散文、全部源碼)與
  **sealed/**(模型不可見:oracle 測試,執行結束後才注入 workdir 再跑驗收)。
- 這測的是「從規格與呼叫點推導正確行為」,不是「對測試擬合」——v1 飽和的根源被移除。
- 公平性:兩臂拿到逐字相同的 shipped 內容;sealed 絕不出現在任何臂的 context。

## 任務(3 題)

| ID | 類型 | shipped | sealed 抓什麼 |
|---|---|---|---|
| T5 | 跨模組功能(預約過期釋放庫存) | 4 模組源碼 + 基本功能測試 | 未過期者不受影響、重複過期冪等、釋放後報表一致 |
| T6 | 曖昧規格(定價規則散文) | spec.md + 明顯案例測試 | 邊界解讀:恰好門檻、折扣加法疊加、四捨五入、免運以折後計 |
| T7 | 修一處不破三處(ID 正規化) | 失敗測試 + 三個依賴模組源碼 | 鄰居模組的既有行為(legacy 零去除、冪等、空值例外) |

## 臂與規模

- **主對照:裸 Opus vs kit 化 Opus**(H2 的兩臂),每格 3 run → 3×2×3 = 18 runs。
- 裸 Fable 臂列為第二波(僅當 kit 顯示增量時加測,作為天花板參照)。
- 預估成本 $15-25;kit 臂允許自行路由(kit as deployed)。

## 指標與判定(預註冊)

- **主要**:密封驗收通過率(每格 3 run 的通過數)。
- **次要**:成本、時間、scope(變更檔數)、shipped 測試通過但 sealed 失敗的「假完成率」。
- H2 成立:kit 通過率 − 裸 Opus 通過率 ≥ 2/9(至少多過兩格)。
- 兩臂皆低(<4/9):任務帶過難或 fixture 缺陷,如實回報,不下 kit 結論。
- 兩臂皆高(≥8/9):難度仍不足,誠實承認並停止在此帶加測。
- **不得為遷就結果改動本判定**。

## 已知限制

- fixture 仍為合成小型專案(百行級),不代表萬行 repo;sealed 測試由 kit 作者設計,
  存在「出題者偏誤」風險——緩解:題目與 sealed 於執行前全部 commit(本檔與 fixtures 同批),
  歡迎外部檢視題目公平性。
